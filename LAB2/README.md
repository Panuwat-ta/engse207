## Architectural Drivers Document : Social Media Platform
---
### ส่วนที่ 1: System Overview 

### 1. System Description

Social Media Platform เป็นแพลตฟอร์มเครือข่ายสังคมออนไลน์ขนาดเล็ก (Niche Social Network) ที่ถูกออกแบบมาเพื่อสร้างพื้นที่คอมมูนิตี้เฉพาะกลุ่ม (Interest-based Community) เช่น กลุ่มศิษย์เก่า, ชมรมถ่ายภาพ, หรือองค์กรภายใน โดยมุ่งเน้นความเรียบง่าย เป็นส่วนตัว และลดการรบกวนจากอัลกอริทึมโฆษณาที่มากเกินไปเหมือนในแพลตฟอร์มขนาดใหญ่

ระบบนี้ช่วยแก้ปัญหา "Information Overload" หรือการได้รับข้อมูลข่าวสารที่ไม่เกี่ยวข้องมากเกินไป และปัญหาเรื่องความเป็นส่วนตัวของข้อมูล โดยระบบจะอนุญาตให้ผู้ใช้งานสร้างโปรไฟล์ โพสต์ข้อความ รูปภาพ และวิดีโอ รวมถึงการสร้างกลุ่มย่อย (Groups) เพื่อแลกเปลี่ยนความคิดเห็นได้อย่างอิสระภายใต้การดูแลของ Admin ในแต่ละกลุ่ม

ผู้ใช้งานหลักของระบบคือ สมาชิกในคอมมูนิตี้นั้นๆ ที่ต้องการพื้นที่ปลอดภัยในการแลกเปลี่ยนความรู้และประสบการณ์ ในขณะที่ผู้ดูแลระบบ (Admin) สามารถบริหารจัดการเนื้อหาและสมาชิกเพื่อให้สังคมออนไลน์นี้น่าอยู่และเป็นระเบียบเรียบร้อย


### 2. Target Users
- General Users (สมาชิกทั่วไป): ผู้ใช้งานที่ต้องการโพสต์เนื้อหา ติดตามเพื่อน และร่วมพูดคุยในกลุ่ม
- Group Moderators (ผู้ดูแลกลุ่ม): ผู้ใช้งานที่มีสิทธิ์จัดการเนื้อหาและสมาชิกภายในกลุ่มเฉพาะที่ตนเองดูแล
- System Administrators (ผู้ดูแลระบบสูงสุด): ทีมงานผู้ดูแลภาพรวมของระบบ ความปลอดภัย และจัดการบัญชีผู้ใช้ทั้งหมด

### 3. Key Features (Top 10)
- User Profiles: สร้างและแก้ไขข้อมูลส่วนตัว รูปโปรไฟล์ และประวัติโดยย่อ

- News Feed: แสดงรายการโพสต์จากเพื่อนและกลุ่มที่ติดตาม เรียงตามลำดับเวลา (Chronological Order)

- Content Posting: รองรับการโพสต์ข้อความ รูปภาพ (Multiple Uploads) และวิดีโอสั้น

- Interaction System: ระบบกดถูกใจ (Like), แสดงอารมณ์ (Reactions) และคอมเมนต์ (Comments)

- Groups & Communities: สร้างกลุ่มสาธารณะ (Public) และกลุ่มส่วนตัว (Private) สำหรับหัวข้อเฉพาะ

- Friend/Follow System: ระบบขอเป็นเพื่อน (Friend Request) หรือกดติดตาม (Follow) เพื่อดูเนื้อหา

- Real-time Notifications: แจ้งเตือนเมื่อมีการกดไลก์ คอมเมนต์ หรือมีโพสต์ใหม่ในกลุ่มที่สนใจ

- Direct Messaging (Chat): แชทส่วนตัวระหว่างผู้ใช้งานแบบ 1-ต่อ-1

- Search & Discovery: ค้นหาเพื่อน กลุ่ม หรือโพสต์ผ่าน Hashtag (#)

- Privacy Controls: ตั้งค่าความเป็นส่วนตัวของโพสต์ (Public, Friends Only, Private)

### ส่วนที่ 2: Functional Requirements

### User Management
- FR-01: ผู้ใช้สามารถลงทะเบียน (Register) ด้วย Email หรือเบอร์โทรศัพท์ และยืนยันตัวตนผ่าน OTP ได้

- FR-02: ผู้ใช้สามารถเข้าสู่ระบบ (Login) และออกจากระบบ (Logout) ได้อย่างปลอดภัย

- FR-03: ผู้ใช้สามารถกู้คืนรหัสผ่าน (Reset Password) ผ่านทาง Email ได้กรณีลืมรหัสผ่าน

- FR-04: ผู้ใช้สามารถแก้ไขข้อมูลส่วนตัว (Edit Profile) เช่น ชื่อ, รูปภาพ, Bio และเปลี่ยนรหัสผ่านได้

### Core Features (Content & Feed)
- FR-05: ผู้ใช้สามารถสร้างโพสต์ (Create Post) ที่ประกอบด้วยข้อความ รูปภาพ (สูงสุด 10 รูป) หรือวิดีโอ (ความยาวไม่เกิน 2 นาที) ได้

- FR-06: ผู้ใช้สามารถแก้ไข (Edit) หรือลบ (Delete) โพสต์ของตนเองได้

- FR-07: ระบบต้องแสดงหน้า News Feed ที่รวบรวมโพสต์จากเพื่อนและกลุ่มที่ผู้ใช้ติดตาม โดยเรียงจากใหม่ไปเก่า

- FR-08: ผู้ใช้สามารถค้นหาโพสต์ด้วย Keyword หรือ Hashtag ได้

### Interaction & Social
- FR-09: ผู้ใช้สามารถกดแสดงความรู้สึก (Like/React) บนโพสต์และคอมเมนต์ได้

- FR-10: ผู้ใช้สามารถเขียนความคิดเห็น (Comment) และตอบกลับความคิดเห็น (Reply) ได้

- FR-11: ผู้ใช้สามารถกด Share โพสต์ไปยังหน้า Feed ของตนเองได้

- FR-12: ผู้ใช้สามารถส่งคำขอเป็นเพื่อน (Send Friend Request), ตอบรับ (Accept), หรือปฏิเสธ (Decline) ได้

### Notification
- FR-13: ระบบต้องแจ้งเตือน (Push Notification/In-app) เมื่อมีคนมาคอมเมนต์หรือกดไลก์โพสต์ของผู้ใช้

- FR-14: ระบบต้องแจ้งเตือนเมื่อมีคำขอเป็นเพื่อนใหม่

### Reporting & Moderation
- FR-15: ผู้ใช้สามารถกดรายงาน (Report) โพสต์หรือผู้ใช้ที่ทำผิดกฎชุมชนได้

- FR-16: Admin สามารถดูรายการที่ถูก Report และดำเนินการลบโพสต์ (Remove Post) หรือแบนผู้ใช้ (Ban User) ชั่วคราว/ถาวรได้

### ส่วนที่ 3: Quality Attributes & Scenarios
#### QA-1: Performance (ประสิทธิภาพ)
#### Scenario: ผู้ใช้งานจำนวนมากเข้าใช้งานหน้า News Feed พร้อมกันในช่วงเวลาเร่งด่วน

| ส่วน               | รายละเอียด                                                  |
|-------------------|-----------------------------------------------------------|
| **Source**        | ผู้ใช้งานจำนวนมาก (Users)                                    |
| **Stimulus**      | ทำการรีเฟรชหน้า News Feed พร้อมกัน 1,000 คน ภายใน 1 วินาที      |
| **Artifact**      | ระบบ News Feed Service                                    |
| **Environment**   | ช่วงเวลา Peak Load ภายใต้การทำงานปกติ                         |
| **Response**      | ระบบประมวลผลและส่งคืนรายการโพสต์ล่าสุดให้ผู้ใช้ทุกคน                |
| **Response Measure** | Latency เฉลี่ยต้องไม่เกิน 2 วินาที (95th percentile)           |
##### คำอธิบาย: ระบบต้องมีการทำ Caching (เช่น Redis) เพื่อลดการดึงข้อมูลจาก Database โดยตรง และรองรับการดึงข้อมูลพร้อมกันได้รวดเร็วเพื่อให้ประสบการณ์ใช้งานลื่นไหล

### QA-2: Scalability (ความสามารถในการขยายตัว)
### Scenario: จำนวนผู้ใช้งานเพิ่มขึ้นอย่างรวดเร็วจากการโปรโมท
| ส่วน                 | รายละเอียด                                                                              |
|---------------------|---------------------------------------------------------------------------------------|
| **Source**          | ทีมการตลาด / ผู้ใช้งานใหม่                                                                 |
| **Stimulus**        | จำนวนผู้ใช้งานเพิ่มขึ้นจาก 10,000 เป็น 50,000 คน ภายใน 1 สัปดาห์                                |
| **Artifact**        | Server Infrastructure (Backend & Database)                                             |
| **Environment**     | ระบบ Cloud Environment (Production)                                                    |
| **Response**        | ระบบต้องสามารถรองรับ Load ที่เพิ่มขึ้นได้โดยไม่ล่ม (Auto-scaling)                                 |
| **Response Measure**| CPU Usage ต้องไม่เกิน 80% และระบบสามารถ Scale out เพิ่มจำนวน Instance ได้อัตโนมัติภายใน 5 นาที |
##### คำอธิบาย: ระบบต้องออกแบบสถาปัตยกรรม (Architecture) ที่รองรับการขยายตัวในแนวนอน (Horizontal Scaling) โดยใช้บริการ Cloud Auto-scaling Group และแยกฐานข้อมูลสำหรับ Read/Write หากจำเป็น เพื่อรองรับปริมาณ Traffic ที่ผันผวน

### QA-3: Security (ความปลอดภัย)
### Scenario: ผู้ไม่ประสงค์ดีพยายามเจาะระบบด้วย SQL Injection
| ส่วน                 | รายละเอียด                                                             |
|---------------------|----------------------------------------------------------------------|
| **Source**          | Hacker / ผู้ไม่ประสงค์ดี                                                  |
| **Stimulus**        | ใส่โค้ด SQL Injection เข้ามาในช่องค้นหาหรือช่อง Login                       |
| **Artifact**        | Database / API Gateway                                               |
| **Environment**     | ระบบที่เปิดใช้งานจริง (Online)                                             |
| **Response**        | ระบบตรวจสอบ Input Validation, ปฏิเสธคำสั่ง, และบันทึก Log การโจมตี         |
| **Response Measure**| ข้อมูลไม่รั่วไหล 0% (Zero Data Leakage) และบัญชีผู้ใช้ต้องไม่ถูกเข้าถึงโดยมิชอบ     |
##### คำอธิบาย: ความปลอดภัยเป็นสิ่งสำคัญสูงสุด ระบบต้องปฏิบัติตามมาตรฐาน OWASP Top 10 มีการตรวจสอบข้อมูลนำเข้า (Input Validation) อย่างเคร่งครัด เข้ารหัสข้อมูลที่ละเอียดอ่อน (Encryption at rest & in transit) และมีการจัดการ Session token อย่างรัดกุม

### QA-4: Availability (ความพร้อมใช้งาน)
### Scenario: Server ตัวหลักเกิดความเสียหาย (Crash)
| ส่วน                 | รายละเอียด                                                                 |
|---------------------|--------------------------------------------------------------------------|
| **Source**          | ฮาร์ดแวร์ขัดข้อง หรือ ซอฟต์แวร์ Bug                                             |
| **Stimulus**        | API Server ตัวหลักหยุดทำงานกะทันหัน                                          |
| **Artifact**        | Load Balancer / Backup Server                                             |
| **Environment**     | Production Environment                                                    |
| **Response**        | Load Balancer ตรวจจับความผิดปกติและ Redirect Traffic ไปยัง Server สำรองทันที   |
| **Response Measure**| Downtime ต้องไม่เกิน 1 นาที (99.9% Availability per month)                    |
##### คำอธิบาย: ระบบต้องมีการทำ Redundancy ในทุกชั้น (Web Server, Database) เพื่อป้องกัน Single Point of Failure และมีแผน Disaster Recovery (DR) เพื่อให้มั่นใจว่าบริการจะกลับมาทำงานได้เร็วที่สุดหากเกิดเหตุขัดข้องร้ายแรง

### ส่วนที่ 4: Constraints
### Technical Constraints
- Mobile-First: ต้องพัฒนาในรูปแบบ Cross-platform mobile app (Flutter หรือ React Native) เพื่อรองรับทั้ง iOS และ Android เป็นลำดับแรก

- Cloud Provider: ต้องใช้งานบน Cloud ของ AWS หรือ Google Cloud Platform เท่านั้น เพื่อความง่ายในการ Scale

- Database: ต้องใช้ Relational Database (PostgreSQL) สำหรับข้อมูลผู้ใช้ และ NoSQL (MongoDB) สำหรับเก็บ Activity Logs และ Feed

### Time Constraints
- MVP Release: ต้องเปิดตัวเวอร์ชันทดลอง (Beta) ภายใน 4 เดือนหลังจากเริ่มโครงการ

- User Testing: ต้องมีระยะเวลา Testing อย่างน้อย 2 สัปดาห์ก่อน Launch จริง

### Budget Constraints
- Development Cost: งบประมาณสำหรับการพัฒนาเฟสแรกไม่เกิน 2,000,000 บาท

- Operation Cost: ค่าใช้จ่าย Cloud Server รายเดือนต้องไม่เกิน 15,000 บาท ในช่วง 6 เดือนแรก

### Legal/Policy Constraints
- PDPA Compliance: ระบบต้องปฏิบัติตาม พ.ร.บ. คุ้มครองข้อมูลส่วนบุคคล (PDPA) ของไทยอย่างเคร่งครัด (มี Consent Form, สิทธิ์ในการลบข้อมูล)

- Content Policy: ต้องมีระบบตรวจจับภาพอนาจารหรือเนื้อหาผิดกฎหมายเบื้องต้น (เช่น ใช้ AI ตรวจจับภาพโป๊เปลือย)

### ส่วนที่ 5: Assumptions
1. Internet Connectivity: ผู้ใช้งานมีความเชื่อมต่ออินเทอร์เน็ตที่เสถียร (4G/5G/Wi-Fi) ตลอดการใช้งาน (ระบบไม่ได้ออกแบบมาให้ทำงาน Offline เต็มรูปแบบ)

2. User Device: ผู้ใช้งานส่วนใหญ่ (90%) ใช้สมาร์ทโฟนที่ผลิตในปี 2020 ขึ้นไป

3. Content Volume: ขนาดเฉลี่ยของรูปภาพที่อัปโหลดจะไม่เกิน 5 MB ต่อรูป และวิดีโอไม่เกิน 50 MB

4. Language: เนื้อหาหลักในระบบจะเป็นภาษาไทยและภาษาอังกฤษ

5. Admins: จะมีเจ้าหน้าที่ Admin (Human Moderator) คอยตรวจสอบ Report ภายใน 24 ชั่วโมง โดยยังไม่ได้ใช้ระบบ AI Moderator เต็มรูปแบบในเฟสแรก

### ส่วนที่ 6: Priority & Trade-offs
### Quality Attributes Priority
| Rank  | Quality Attribute | เหตุผล                                                                          |
|-------|-------------------|--------------------------------------------------------------------------------|
| **1** | Security          | ข้อมูลส่วนตัวและการแชทเป็นเรื่องอ่อนไหว หากข้อมูลรั่วไหลจะทำลายความน่าเชื่อถือทันที (Critical) |
| **2** | Performance       | Social Network ต้องลื่นไหล หากโหลดช้าเกิน 3 วินาที ผู้ใช้จะเลิกเล่น (High)                |
| **3** | Availability      | ระบบต้องเข้าถึงได้ตลอดเวลา แต่หากล่มช่วงสั้นๆ อาจพอรับได้มากกว่าข้อมูลหลุด (High)           |
| **4** | Usability         | ต้องใช้งานง่าย แต่ผู้ใช้สามารถเรียนรู้ได้หากฟีเจอร์มีประโยชน์ (Medium)                        |
| **5** | Scalability       | สำคัญในระยะยาว แต่ช่วงแรกจำนวนผู้ใช้ยังจำกัด สามารถปรับจูนภายหลังได้ (Medium)             |
| **6** | Maintainability   | สำคัญต่อทีม Dev แต่ไม่กระทบ End-user โดยตรงในระยะสั้น (Low)                          |

### Trade-offs Analysis
### Trade-off #1: Security vs. Performance
- คำอธิบาย: การเพิ่มมาตรการความปลอดภัยขั้นสูง เช่น การเข้ารหัสข้อมูลทุกจุด (End-to-End Encryption ในแชท), การตรวจสอบสิทธิ์แบบ Multi-factor, และการ Validate Input อย่างละเอียดทุก Request จะทำให้ระบบต้องใช้เวลาประมวลผลเพิ่มขึ้น (Overhead) ซึ่งส่งผลให้ Response Time ช้าลง

- Decision: เลือก Security เป็นหลักในส่วนของข้อมูลส่วนตัวและแชท แม้จะช้าลงเล็กน้อย (ระดับ Milliseconds) แต่ยอมรับได้ เพราะความเชื่อมั่นสำคัญที่สุด ส่วนหน้า Feed หรือรูปภาพสาธารณะ ให้ลดระดับ Encryption ลงบางส่วนหรือใช้ Caching ช่วยเพื่อเน้น Performance

### Trade-off #2: Usability vs. Features (Completeness)
- คำอธิบาย: การใส่ฟีเจอร์จำนวนมากให้ครบถ้วน (เช่น แต่งรูปได้ละเอียด, ตั้งค่า Privacy ซับซ้อน) จะทำให้หน้าจอดูรกและใช้งานยากสำหรับผู้เริ่มต้น (Low Usability) เทียบกับการทำหน้าจอให้คลีนๆ แต่อาจขาดบางฟีเจอร์ไป

- Decision: เลือก Usability ในช่วงเปิดตัว (MVP) โดยตัดฟีเจอร์ที่ซับซ้อนออกไปก่อน ให้เหลือเฉพาะ Core Feature ที่ใช้งานง่าย เพื่อให้ผู้ใช้ใหม่ไม่รู้สึกสับสนและเกิดการ Adoption ได้ง่าย แล้วค่อยเพิ่มฟีเจอร์ Advance ในเมนู "More" ภายหลัง
